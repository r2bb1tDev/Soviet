# Криптография в Soviet Messenger

## Принципы

1. **Проверенные примитивы** — используем только стандартные, аудированные алгоритмы
2. **Rust crates** — криптография через надёжные Rust-библиотеки (sodiumoxide, chacha20poly1305, ed25519-dalek, curve25519-dalek)
3. **Zero-knowledge архитектура** — сервер/relay никогда не видит содержимое
4. **Perfect Forward Secrecy** — каждое сообщение использует новые эфемерные ключи
5. **Подписание всех сообщений** — защита от подделки и MITM-атак

## Идентичность пользователя

### Генерация ключей

При первом запуске генерируется **мастер-пара ключей Ed25519**:

```
Ed25519 Keypair (32 bytes):
├── Приватный ключ (32 bytes)  →  Хранится в защищённом хранилище ОС
└── Публичный ключ (32 bytes)  →  Base58-кодирование, это ваш "номер"

Производный X25519 ключ для шифрования:
└── Конвертируется из Ed25519 публичного ключа для ECDH-операций
```

### Публичный ключ как идентификатор

Публичный ключ в формате Base58 — это **уникальный и неизменяемый** идентификатор пользователя. Никнейм — лишь отображаемое имя, которое может меняться.

```
Публичный ключ Ed25519:    5KYZdUEo39C1shXXPRJ4FxHPCcMHbhcVVxjBkzmV9Mfx (52 символа)
Base64-декодирование:      32 bytes
```

### Custom ID (опционально)

Дополнительный идентификатор для удобства:
- 3-10 символов (буквы, цифры, подчёркивание)
- Формат: `@myid`
- Хранится в БД, не используется для криптографии
- Видно в QR-коде и share card

## Шифрование личных сообщений (E2E)

### Алгоритм ECDH + HKDF + ChaCha20-Poly1305

Используется схема **как в Signal Protocol (упрощённая версия)**:

```
┌─────────────────────────────────────────────────────────────┐
│ 1. ECDH Key Agreement (Elliptic Curve Diffie-Hellman)       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Отправитель генерирует эфемерную X25519 пару:             │
│  (ephemeral_sk, ephemeral_pk) ← X25519.generate()           │
│                                                             │
│  Вычисляет общий секрет:                                    │
│  shared_secret = X25519(ephemeral_sk, recipient_pk)         │
│    где recipient_pk — публичный X25519 получателя           │
│         (конвертирован из его Ed25519)                      │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ 2. Key Derivation (HKDF-SHA256)                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  encryption_key = HKDF-SHA256(                              │
│    ikm = shared_secret (32 bytes),                          │
│    salt = sender_pk || recipient_pk (64 bytes),             │
│    info = "soviet-messenger-v1" (ASCII),                    │
│    output_length = 32 bytes                                 │
│  )                                                          │
│                                                             │
│  Функция HKDF раскладывает ключ на Extract и Expand:       │
│  - Extract: PRK = HMAC-SHA256(salt, ikm)                    │
│  - Expand: OKM = HMAC-SHA256(PRK, info || counter)          │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ 3. Encryption (ChaCha20-Poly1305 AEAD)                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  nonce = random_bytes(12)  ← random, no repeating!          │
│                                                             │
│  ciphertext = ChaCha20Poly1305.encrypt(                     │
│    key = encryption_key (32 bytes),                         │
│    nonce = nonce (12 bytes),                                │
│    plaintext = message_bytes,                               │
│    aad = timestamp || content_type (Additional Auth Data)   │
│  )                                                          │
│                                                             │
│  Результат:                                                │
│  - Зашифрованное сообщение                                  │
│  - Аутентификационный тег (16 bytes)                        │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ 4. Signing (Ed25519)                                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  message_to_sign = ephemeral_pk || nonce || ciphertext      │
│  (все в бинарном формате)                                   │
│                                                             │
│  signature = Ed25519.sign(                                  │
│    secret_key = sender_private_key,                         │
│    message = message_to_sign                                │
│  )                                                          │
│                                                             │
│  Подпись: 64 bytes                                          │
└─────────────────────────────────────────────────────────────┘
```

### Итоговый пакет сообщения

```json
{
  "version": 1,
  "sender_pk": "5KYZdUEo39C1shXXPRJ4FxHPCcMHbhcVVxjBkzmV9Mfx",
  "ephemeral_pk": "3dq9KJzV5qpL8xXXvV4h1YXKLMoY6zS1nNPp8rRqT2V",
  "nonce": "b2ZZNE5YZJjkM2RxFgH5",
  "ciphertext": "RmOPm7vvXXUPLLmDxKL...",
  "signature": "KpX5kZ7vMV...",
  "timestamp": 1700000000,
  "content_type": "text"
}
```

**Размер:**
- Эфемерный публичный ключ: 32 bytes (Base58: ~44 символа)
- Nonce: 12 bytes (Base64: ~16 символов)
- Ciphertext: зависит от размера сообщения + 16 bytes тега
- Signature: 64 bytes (Base64: ~88 символов)
- Итого: ~200-500 bytes для сообщения 10-100 bytes

### Расшифровка получателем

```
┌─────────────────────────────────────────────────────────────┐
│ 1. Верификация подписи (Ed25519.verify)                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  message_to_verify = ephemeral_pk || nonce || ciphertext    │
│                                                             │
│  valid = Ed25519.verify(                                    │
│    public_key = sender_pk,                                  │
│    message = message_to_verify,                             │
│    signature = signature                                    │
│  )                                                          │
│                                                             │
│  Если invalid → REJECT сообщение                            │
│  (защита от подделки и MITM)                                │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ 2. ECDH (вычисление общего секрета)                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  shared_secret = X25519(my_secret_key, ephemeral_pk)        │
│                                                             │
│  Результат: то же значение, что и у отправителя!           │
│  (ECDH гарантирует это)                                     │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ 3. HKDF (вывод ключа)                                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  encryption_key = HKDF(shared_secret, salt, info)           │
│  (то же самое, что у отправителя, потому что salt/info      │
│   одинаковые)                                               │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ 4. Расшифровка (ChaCha20Poly1305.decrypt)                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  plaintext = ChaCha20Poly1305.decrypt(                      │
│    key = encryption_key,                                    │
│    nonce = nonce,                                           │
│    ciphertext = ciphertext,                                 │
│    aad = timestamp || content_type                          │
│  )                                                          │
│                                                             │
│  Если расшифровка не удалась → REJECT                       │
│  (защита от повреждения и MITM)                             │
└─────────────────────────────────────────────────────────────┘
```

## Групповые чаты

Для групп используется **симметричный групповой ключ**:

```
┌──────────────────────────────────────────────────────┐
│ Создание группы                                      │
├──────────────────────────────────────────────────────┤
│                                                      │
│ 1. Создатель генерирует случайный GroupKey (32 bytes)│
│ 2. Каждому участнику групповой ключ отправляется    │
│    в зашифрованном виде (через E2E схему выше)      │
│ 3. Участник сохраняет GroupKey в БД                 │
└──────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────┐
│ Отправка сообщения в группу                          │
├──────────────────────────────────────────────────────┤
│                                                      │
│ ciphertext = ChaCha20Poly1305.encrypt(               │
│   key = GroupKey,        ← не эфемерный!            │
│   nonce = random (12 bytes),                         │
│   plaintext = message,                               │
│   aad = group_id || timestamp                        │
│ )                                                    │
│                                                      │
│ signature = Ed25519.sign(sender_sk, ciphertext)      │
│ (подпись для верификации отправителя в группе)      │
└──────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────┐
│ Ротация ключа при добавлении/удалении участника     │
├──────────────────────────────────────────────────────┤
│                                                      │
│ Когда участник добавлен или удалён:                 │
│                                                      │
│ 1. Создатель генерирует новый GroupKey              │
│ 2. Отправляет старому GroupKey всем старым членам   │
│    (зашифрованное уведомление о замене ключа)       │
│ 3. Отправляет новый GroupKey новым членам           │
│    (через E2E шифрование)                           │
│                                                      │
│ Удалённый участник больше не может расшифровать     │
│ новые сообщения!                                    │
└──────────────────────────────────────────────────────┘
```

## Nostr события

Nostr использует **Ed25519 для подписей**, но иначе:

```json
{
  "id": "<32-byte hex-encoded SHA256 hash>",
  "pubkey": "<32-byte hex sender pubkey>",
  "created_at": 1700000000,
  "kind": 42,
  "tags": [
    ["e", "<event-id>", "<relay-url>"],
    ["p", "<pubkey>"]
  ],
  "content": "<encrypted message>",
  "sig": "<64-byte hex signature>"
}
```

**Виды событий в Soviet:**

- `kind=40` — Create channel metadata
- `kind=41` — Update channel metadata
- `kind=42` — Channel message
- `kind=5` — Delete event

**Шифрование контента:**

NIP-04 (старый стандарт) или NIP-44 (новый стандарт):
```
content = base64(
  ChaCha20Poly1305.encrypt(
    key = HKDF(shared_secret, ...),
    nonce = random(12),
    plaintext = message_json
  )
)
```

В v1.0 Soviet каналы **публичные** (не шифруются), только приватные сообщения шифруются.

## Защита от атак

### Защита от MITM (Man-in-The-Middle)

**Проблема:** Если атакующий перехватит публичный ключ контакта и подменит его своим, он сможет расшифровать все сообщения.

**Решение: Safety Numbers (как в Signal)**

```
safety_number = SHA256(sort(my_pubkey, contact_pubkey))
             → отображается как 12 групп по 5 цифр
             → пример: 12345 67890 12345 67890 12345 67890
                       12345 67890 12345 67890 12345 67890

Два пользователя сравнивают цифры лично или по другому каналу.
Если совпадают → атаки MITM нет.
```

### Защита от Replay-атак

Каждое сообщение содержит:
- `timestamp` — время отправки (Unix timestamp)
- `nonce` — случайное число, никогда не повторяется

Сообщение с одинаковым `(sender_pk, ephemeral_pk, nonce)` отбраковывается как дублика.

### Защита от Forging

Каждое сообщение подписано `Ed25519.sign()`. Если подпись неверна → сообщение отбрасывается.

### Защита от Decryption Failures

ChaCha20-Poly1305 включает аутентификационный тег (16 bytes). Если тег неверен → расшифровка падает с ошибкой. Сообщение не обрабатывается.

## Хранение ключей (OS Keystore)

| ОС | Технология | Преимущества |
|----|-----------|-------------|
| **Windows** | DPAPI (Data Protection API) + Credential Manager | Шифрование на уровне ОС, привязано к профилю пользователя |
| **macOS** | Keychain | iCloud Keychain sync (опционально), защита от read/debug |
| **Linux** | libsecret / KWallet / fallback файл с шифрованием | Зависит от DE (GNOME/KDE) |

**Гарантии:**
- ✅ Приватный ключ никогда не скопируется в логи
- ✅ Приватный ключ никогда не скопируется в memory dump
- ✅ Приватный ключ не экспортируется без явного действия пользователя
- ✅ Заминованные ключи стираются (wipe from memory при завершении)

## Аудит криптографии

Весь криптографический код находится в:

```
src-tauri/src/crypto/
├── mod.rs
├── keys.rs       # Ed25519/X25519 генерация, конверсия
├── cipher.rs     # ChaCha20-Poly1305, HKDF
└── signing.rs    # Ed25519 подпись/верификация
```

**Что проверить при аудите:**

1. ✅ ECDH правильно использует X25519
2. ✅ HKDF использует SHA-256, а не MD5
3. ✅ Nonce никогда не повторяется
4. ✅ Приватные ключи стираются из памяти (zeroize)
5. ✅ Timing attacks не возможны
6. ✅ AEAD тег проверяется перед использованием plaintext

**Известные ограничения:**

- Нет поддержки Forward Secrecy в групповых чатах (GroupKey не меняется за сессию)
- Relay-серверы для Nostr могут видеть метаданные (когда/откуда соединение)
- Нет Tor-поддержки для анонимизации трафика
