## Модель безопасности

### Что Soviet защищает

✅ **Конфиденциальность сообщений:**
- E2E шифрование прямых сообщений (ChaCha20-Poly1305 через ECDH)
- Группаовой ключ для групповых чатов (переротируется при добавлении/удалении)
- Только получатель может расшифровать сообщение

✅ **Целостность сообщений:**
- Ed25519 подписи на всех сообщениях
- AEAD (ChaCha20-Poly1305) с аутентификационным тегом
- Защита от изменения сообщения

✅ **Аутентификация:**
- Публичный ключ как уникальный идентификатор
- Подпись отправителя верифицируется при получении
- Защита от подделки сообщений

✅ **Защита от MITM (Man-in-the-Middle):**
- Safety Numbers (как в Signal) — верификация публичного ключа
- QR-коды для первого добавления контакта
- Возможность сравнить fingerprint лично

✅ **Perfect Forward Secrecy:**
- Каждое сообщение использует новую эфемерную X25519 пару
- Даже если скомпрометирован один долгосрочный ключ, старые сообщения остаются зашифрованными

✅ **Локальное хранилище:**
- Приватные ключи хранятся в защищённом хранилище ОС (DPAPI/Keychain/libsecret)
- База данных локальна, не синхронизируется

### Что Soviet НЕ защищает

❌ **Метаданные (в интернет-режиме):**
- Relay серверы видят: кто отправляет, кому, когда
- Может быть использовано для определения паттернов коммуникации
- Рекомендация: используйте LAN-режим для максимальной приватности

❌ **Шифрование БД:**
- SQLite в v1.0 **не зашифрована**
- Если устройство украдено → база видна на диске
- Планируется в v2.0 (SQLCipher)

❌ **Анонимность:**
- Публичный ключ привязан к вашей личности
- Relay серверы (Nostr) знают ваш IP адрес
- Для анонимности используйте VPN/Tor (планируется в v3.0)

❌ **Защита от компрометации устройства:**
- Если устройство хакнуто → приватные ключи видны
- Защита на уровне ОС: используйте биометрию, пароли

## Уязвимости и ответственное раскрытие

### Как сообщить об уязвимости

**НИКОГДА не публикуйте уязвимости безопасности в публичных Issues!**

Вместо этого:

1. **Напишите на email:** security@[to-be-configured]
   - (Заполняется при публикации проекта)

2. **Укажите в письме:**
   - Описание уязвимости (техническое)
   - Шаги воспроизведения (пошагово)
   - Потенциальное воздействие (что может произойти)
   - Версия Soviet (где найдена)
   - Ваша контактная информация (опционально, но желательно)
   - GPG публичный ключ (если хотите зашифрованный ответ)

3. **Дождитесь ответа:**
   - Мы ответим в течение **72 часов**
   - Подтвердим получение и начало работы над патчем

### График исправления

| Тяжесть | Срок |
|---------|------|
| **Критическая** (RCE, полная компрометация ключей) | 1-2 недели |
| **Высокая** (E2E шифрование нарушено) | 2-4 недели |
| **Средняя** (DoS, утечка метаданных) | 4-8 недель |
| **Низкая** (UI/UX, minor bugs) | Best effort |

### Наши обязательства

- ✅ **Оперативность** — немедленный ответ исследователю
- ✅ **Прозрачность** — информировать о статусе работ
- ✅ **Благодарность** — публично поблагодарить (если желает)
- ✅ **Кредит** — указать исследователя в Release Notes
- ✅ **Безопасность** — не преследовать добросовестных исследователей

### Ответственное раскрытие (Coordinated Disclosure)

**Процесс:**

```
День 1: Исследователь отправляет отчёт
        ↓
День 2-3: Наша команда подтверждает уязвимость
        ↓
День 3-30: Разработка и тестирование патча
        ↓
День 30: Выпуск патча + Release Notes с CVE ID
        ↓
День 30: Публичное раскрытие (координированное)
```

**Embargo (эмбарго):**
- Исследователь согласен **не публиковать** подробности до выпуска патча
- После выпуска → полное раскрытие на благо сообщества
- CVE ID выдаётся для отслеживания

## Известные ограничения безопасности (в v1.0)

### 1. SQLite не зашифрована

```
Риск: Если устройство украдено или скомпрометировано
      → вся история сообщений видна в cleartext

Статус: ПЛАНИРУЕТСЯ В v2.0 (SQLCipher)

Текущая защита:
└─ Приватные ключи в защищённом хранилище ОС (DPAPI/Keychain/libsecret)
└─ БД доступна только текущему пользователю ОС
```

### 2. Relay метаданные

```
Риск: Nostr relay видит IP адрес, время, паттерны активности

Статус: ПЛАНИРУЕТСЯ В v3.0 (Tor/VPN support)

Текущая защита:
└─ Используйте LAN-режим для максимальной приватности
└─ Используйте VPN вручную при работе с relay
```

### 3. Нет защиты от SYBIL атак

```
Риск: Злоумышленник может создать тысячи поддельных узлов

Статус: ПЛАНИРУЕТСЯ В v2.0 (reputation система)

Текущая защита:
└─ Верификация контактов через Safety Numbers
└─ QR-коды для первого добавления
```

### 4. LAN-режим требует локальной сети

```
Риск: Если вас в той же сети что и злоумышленник
      → может видеть метаданные

Статус: КОНТРОЛИРУЕТСЯ

Защита:
├─ Все сообщения E2E зашифрованы
├─ Подписи верифицируются
└─ Даже если видит трафик → не может расшифровать
```

## Рекомендации по использованию

### Защита приватного ключа

- ✅ Храните приватный ключ в защищённом хранилище ОС
- ✅ Включите биометрию/пароль на устройстве
- ✅ Регулярно обновляйте ОС и приложение
- ❌ Не экспортируйте приватный ключ без необходимости
- ❌ Не делитесь приватным ключом ни с кем
- ❌ Не сохраняйте приватный ключ в текстовых файлах

### Верификация контактов

- ✅ Верифицируйте Safety Numbers лично или по другому защищённому каналу
- ✅ Используйте QR-коды при первом добавлении контакта
- ✅ Сравните Safety Number вашего контакта с вашим (они должны совпадать)
- ❌ Не доверяйте контактам без верификации в критичных ситуациях

### Безопасность устройства

- ✅ Используйте пароль/биометрию на устройстве
- ✅ Регулярно обновляйте ОС
- ✅ Используйте антивирус (для Windows)
- ✅ Не устанавливайте приложения из неизвестных источников
- ✅ Выключайте Bluetooth/Wi-Fi когда не нужны
- ❌ Не подключайте устройство к неизвестным компьютерам

### Использование LAN-режима

- ✅ LAN-режим максимально приватен (нет relay серверов)
- ✅ Используйте в домашней или офисной сети
- ❌ Не используйте в публичной Wi-Fi без VPN
- ❌ Не доверяйте открытым Wi-Fi сетям

### Использование Nostr-режима

- ✅ Понимайте, что relay видит ваш IP адрес и паттерны активности
- ✅ Используйте VPN при необходимости приватности метаданных
- ❌ Не публикуйте чувствительную информацию в публичных каналах (она видна всем!)
- ❌ Помните, что Nostr — это публичная сеть

## Криптографический аудит

### Проверенные компоненты

- ✅ **Ed25519** — стандартная реализация (sodiumoxide / ed25519-dalek)
- ✅ **X25519** — стандартная реализация (curve25519-dalek)
- ✅ **ChaCha20-Poly1305** — AEAD, используется в TLS 1.3
- ✅ **HKDF-SHA256** — стандартная функция вывода ключей
- ✅ **SHA-256** — криптографический хэш

### Используемые Rust crates

```toml
ed25519-dalek = "2.0"           # Ed25519 (Rust impl)
curve25519-dalek = "4.0"        # X25519
chacha20poly1305 = "0.10"       # AEAD
hkdf = "0.12"                   # HKDF-SHA256
sha2 = "0.10"                   # SHA-256
bs58 = "0.5"                    # Base58 encoding
base64 = "0.21"                 # Base64 encoding
```

**Все crates используют проверенные и аудированные реализации.**
