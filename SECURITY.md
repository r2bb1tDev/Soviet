# Политика безопасности (Security Policy)

## Поддерживаемые версии

| Версия | Статус | Поддержка безопасности |
|--------|--------|------------------------|
| 2.1.x (latest) | ✅ Активная | Все патчи безопасности |
| 2.0.x | ⚠️ Устаревшая | Только критические патчи |
| < 2.0 | ❌ Не поддерживается | Нет |

## Модель безопасности

### Что Soviet защищает

✅ **Конфиденциальность сообщений:**
- E2E шифрование прямых сообщений (ChaCha20-Poly1305 через ECDH X25519)
- Групповой ключ для групповых чатов (ротируется при добавлении/удалении участников)
- Только получатель может расшифровать сообщение

✅ **Целостность сообщений:**
- Ed25519 подписи на всех сообщениях
- AEAD (ChaCha20-Poly1305) с аутентификационным тегом
- Защита от подделки и изменения сообщений

✅ **Аутентификация:**
- Публичный ключ как уникальный идентификатор
- Подпись отправителя верифицируется при каждом получении
- Защита от подделки сообщений

✅ **Защита от MITM:**
- Safety Numbers — верификация публичного ключа
- QR-коды для первого добавления контакта
- Возможность сравнить fingerprint лично

✅ **Perfect Forward Secrecy:**
- Каждое сообщение использует новую эфемерную X25519 пару
- Компрометация долгосрочного ключа не раскрывает старые сообщения

✅ **Шифрование чувствительных полей БД (с v2.0):**
- Nostr приватный ключ хранится в БД в зашифрованном виде (ChaCha20-Poly1305)
- Ключ шифрования хранится в защищённом хранилище ОС (DPAPI / Keychain / libsecret)
- Случайный nonce при каждом шифровании

✅ **Локальное хранилище:**
- Приватные Ed25519/X25519 ключи хранятся исключительно в защищённом хранилище ОС
- База данных локальна, не синхронизируется ни с какими серверами

✅ **Сетевая безопасность:**
- Шум-протокол (Noise Protocol) через libp2p для P2P соединений
- NAT traversal через DCUtR (Direct Connection Upgrade through Relay) — с v2.0
- Relay серверы не видят содержимое зашифрованных сообщений

### Что Soviet НЕ защищает

❌ **Метаданные в интернет-режиме:**
- Nostr relay серверы видят: кто отправляет, кому, когда (но не содержимое)
- Может использоваться для анализа паттернов коммуникации
- Рекомендация: LAN-режим для максимальной приватности

❌ **Полное шифрование файла базы данных:**
- История сообщений в SQLite хранится без шифрования файла
- Nostr приватный ключ зашифрован (с v2.0), но контент чатов — нет
- Если устройство украдено → история переписки видна на диске
- Планируется в v2.2 (SQLCipher)

❌ **Анонимность:**
- Публичный ключ может быть связан с вашей личностью
- Relay серверы (Nostr) знают ваш IP адрес
- Для анонимности — используйте VPN/Tor (планируется в v3.0)

❌ **Защита от компрометации устройства:**
- Если устройство взломано → приватные ключи и история сообщений могут быть доступны
- Защита на уровне ОС: используйте биометрию и надёжные пароли

## Уязвимости и ответственное раскрытие

### Как сообщить об уязвимости

**НИКОГДА не публикуйте уязвимости безопасности в публичных Issues!**

Вместо этого:

1. **Напишите на email:** r2bb1t.dev@gmail.com

2. **Укажите в письме:**
   - Описание уязвимости (техническое)
   - Шаги воспроизведения (пошагово)
   - Потенциальное воздействие (что может произойти)
   - Версия Soviet (где найдена)
   - Ваша контактная информация (опционально, но желательно)
   - GPG публичный ключ (если хотите зашифрованный ответ)

3. **Дождитесь ответа:**
   - Мы ответим в течение **72 часов**
   - Подтвердим получение и начало работы над патчем

### График исправления

| Тяжесть | Срок |
|---------|------|
| **Критическая** (RCE, полная компрометация ключей) | 1–2 недели |
| **Высокая** (E2E шифрование нарушено) | 2–4 недели |
| **Средняя** (DoS, утечка метаданных) | 4–8 недель |
| **Низкая** (UI/UX, minor bugs) | Best effort |

### Наши обязательства

- ✅ **Оперативность** — ответ исследователю в течение 72 часов
- ✅ **Прозрачность** — информировать о статусе работ
- ✅ **Благодарность** — публично поблагодарить (если желает)
- ✅ **Кредит** — указать исследователя в Release Notes
- ✅ **Безопасность** — не преследовать добросовестных исследователей

### Ответственное раскрытие (Coordinated Disclosure)

```
День 1:   Исследователь отправляет отчёт
          ↓
День 2-3: Наша команда подтверждает уязвимость
          ↓
День 3-30: Разработка и тестирование патча
          ↓
День 30:  Выпуск патча + Release Notes с CVE ID
          ↓
День 30+: Публичное раскрытие (координированное)
```

## Известные ограничения безопасности (v2.1)

### 1. Полное шифрование SQLite не реализовано

```
Риск: Если устройство украдено или скомпрометировано
      → история сообщений видна в открытом виде на диске

Статус: ПЛАНИРУЕТСЯ В v2.2 (SQLCipher)

Текущая защита:
└─ Nostr приватный ключ зашифрован в БД (ChaCha20-Poly1305, с v2.0)
└─ Ed25519/X25519 ключи — в защищённом хранилище ОС
└─ БД доступна только текущему пользователю ОС
```

### 2. Relay метаданные

```
Риск: Nostr relay видит IP адрес, время, паттерны активности

Статус: ПЛАНИРУЕТСЯ В v3.0 (Tor/VPN интеграция)

Текущая защита:
└─ Используйте LAN-режим для максимальной приватности
└─ Используйте VPN вручную при работе с relay
```

### 3. Нет защиты от Sybil-атак

```
Риск: Злоумышленник может создать множество поддельных узлов

Статус: ПЛАНИРУЕТСЯ В v2.2 (reputation система)

Текущая защита:
└─ Верификация контактов через Safety Numbers
└─ QR-коды для первого добавления контакта
```

### 4. LAN-режим и публичные сети

```
Риск: В одной сети с атакующим → видны метаданные соединений

Статус: КОНТРОЛИРУЕТСЯ

Защита:
├─ Все сообщения E2E зашифрованы (ChaCha20-Poly1305)
├─ Ed25519 подписи верифицируются при каждом получении
└─ Даже при перехвате трафика — расшифровка невозможна
```

## Рекомендации по использованию

### Защита приватного ключа

- ✅ Не экспортируйте приватный ключ без крайней необходимости
- ✅ Включите биометрию/пароль на устройстве
- ✅ Регулярно обновляйте ОС и приложение
- ❌ Не делитесь приватным ключом ни с кем
- ❌ Не сохраняйте приватный ключ в текстовых файлах

### Верификация контактов

- ✅ Верифицируйте Safety Numbers лично или по другому защищённому каналу
- ✅ Используйте QR-коды при первом добавлении контакта
- ❌ Не доверяйте контактам без верификации в критичных ситуациях

### Безопасность устройства

- ✅ Используйте пароль/биометрию на устройстве
- ✅ Регулярно обновляйте ОС
- ✅ Не подключайте устройство к неизвестным компьютерам
- ❌ Не устанавливайте приложения из неизвестных источников

### Использование LAN-режима

- ✅ LAN-режим максимально приватен (нет relay серверов, нет интернета)
- ✅ Используйте в домашней или корпоративной сети
- ❌ Не используйте в публичной Wi-Fi без VPN

### Использование Nostr-режима

- ✅ Relay видит только зашифрованный blob — содержимое недоступно
- ✅ Используйте VPN при необходимости скрыть IP адрес
- ❌ Не публикуйте чувствительную информацию в публичных каналах
- ❌ Помните: Nostr — публичная сеть, посты в каналах видны всем подписчикам

## Криптографические компоненты

### Используемые алгоритмы

| Компонент | Алгоритм | Назначение |
|-----------|----------|------------|
| Ключи идентичности | Ed25519 | Подпись сообщений, идентификация |
| Обмен ключами | X25519 ECDH | Shared secret для шифрования |
| Шифрование сообщений | ChaCha20-Poly1305 | AEAD E2E шифрование |
| Вывод ключей | HKDF-SHA256 | Ключи из shared secret |
| Хэширование | SHA-256 | Идентификаторы, fingerprints |
| P2P транспорт | Noise Protocol | libp2p соединения |
| Шифрование полей БД | ChaCha20-Poly1305 | Nostr приватный ключ в БД |
| Хранение мастер-ключа | OS Keyring | DPAPI / Keychain / libsecret |

### Используемые Rust crates

```toml
ed25519-dalek       = "2"      # Ed25519 (аудированная реализация)
curve25519-dalek    = "4"      # X25519
chacha20poly1305    = "0.10"   # AEAD шифрование
hkdf                = "0.12"   # HKDF-SHA256
sha2                = "0.10"   # SHA-256
keyring             = "2"      # OS keyring (DPAPI/Keychain/libsecret)
libp2p              = "0.53"   # P2P сеть с Noise Protocol
```

**Все crates используют проверенные и широко применяемые реализации.**
